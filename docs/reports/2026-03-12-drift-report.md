## Critical Issues

### C1: Naur layer enum values in `schemas.ts` are inconsistent with the design contracts, DB migrations, and API type aliases

- **Requirement:** Story 4.1 — generate questions targeting Naur's three layers: "world-to-program mapping, design justification, modification capacity".
- **Expected:** `src/lib/engine/llm/schemas.ts` `NaurLayerSchema` values should match the canonical names established in design doc section 4.1 (database `CHECK` constraint), section 4.4 (API `NaurLayer` type alias), and section 4.6 (LLM response schema):
  - `'world_to_program'`
  - `'design_justification'`
  - `'modification_capacity'`
- **Found:** `NaurLayerSchema` defines three abbreviated values that match none of the above:
  - `'world_mapping'`
  - `'design'`
  - `'modification'`
- **Impact:** This is a three-way mismatch across the system boundary. The database `assessment_questions.naur_layer` column has a `CHECK (naur_layer IN ('world_to_program', 'design_justification', 'modification_capacity'))` constraint (migration `20260309000002_assessment_tables.sql`). The design doc L4 API contract defines `NaurLayer = 'world_to_program' | 'design_justification' | 'modification_capacity'`. The LLM response schema contract in design section 4.6 also uses `world_to_program | design_justification | modification_capacity`. Any question generated by the engine and then persisted to `assessment_questions` will fail the DB `CHECK` constraint at insert time, because the engine produces `'world_mapping'`, `'design'`, `'modification'` but the DB expects `'world_to_program'`, `'design_justification'`, `'modification_capacity'`. The engine will also reject valid LLM responses that return the DB-conformant names, because schema validation in `AnthropicClient.generateStructured` applies `NaurLayerSchema`. This is a latent runtime failure in the integration path that tests cannot yet detect because no persistence layer has been wired up.

## Warnings

### W1: `RelevanceResponseSchema` field name differs from the design doc LLM contract

- **Location:** `src/lib/engine/llm/schemas.ts` line 31 vs design doc section 4.6 (Relevance detection response schema)
- **Issue:** The design contract specifies the relevance response field as `"is_relevant"` (a boolean). The `RelevanceResponseSchema` uses `relevant` (no `is_` prefix). The design doc also uses `is_relevant` as the database column name on `participant_answers`, and the API answers response contract in section 4.4 returns `is_relevant: boolean`. The Zod schema returning `{ relevant, explanation }` diverges from every other layer that expects `{ is_relevant, explanation }`. Any code that reads from the validated response and passes `is_relevant` to the DB insert will need to rename the field, creating a silent mapping gap.
- **Suggested action:** Rename `relevant` to `is_relevant` in `RelevanceResponseSchema` to match the design contract. Update fixtures and mock clients accordingly.

### W2: `QuestionGenerationResponseSchema` does not capture `artefact_quality` or `artefact_quality_note`

- **Location:** `src/lib/engine/llm/schemas.ts` vs design doc section 4.6 (Question generation response schema)
- **Issue:** The design L4 contract specifies that the LLM question generation response includes three top-level fields: `questions[]`, `artefact_quality` (enum), and `artefact_quality_note` (string). The `QuestionGenerationResponseSchema` only validates `questions`. The `artefact_quality` field is functionally important: it is stored on the `assessments` table (`artefact_quality text`), surfaced in FCS results (Story 3.4, 6.2), and is the signal Story 4.1 references as `artefact_quality: code_only`. Omitting it from the Zod schema means the information will be silently lost when the engine processes the LLM response.
- **Suggested action:** Add `artefact_quality` (z.enum for the four values: `code_only`, `code_and_tests`, `code_and_requirements`, `code_requirements_and_design`) and `artefact_quality_note` (z.string()) to `QuestionGenerationResponseSchema`.

### W3: `QuestionSchema` uses `text` and `id` fields; design contract uses `question_text` and `question_number`

- **Location:** `src/lib/engine/llm/schemas.ts` `QuestionSchema` vs design doc section 4.6 LLM response schema and section 4.1 DB schema
- **Issue:** `QuestionSchema` defines `{ id, text, weight, naur_layer, reference_answer }`. The design LLM response schema specifies `{ question_number, naur_layer, question_text, weight, reference_answer }`. The DB column is `question_text` (not `text`). The design uses `question_number` (integer sequence) rather than `id` (string). This is a structural mismatch: the engine will produce questions with field names that do not match what the persistence layer expects. Any mapping code between the engine output and the DB insert will have to rename `text` → `question_text` and convert or discard `id` in favour of `question_number`.
- **Suggested action:** Align `QuestionSchema` with the design contract: use `question_number: z.number().int()` and `question_text: z.string()`. Remove the `id` field (the DB generates its own UUID primary key) or make it optional and unused.

### W4: LLM client default model (`claude-3-5-sonnet-20241022`) differs from design doc specification

- **Location:** `src/lib/engine/llm/client.ts` line 43 vs design doc section 4.6 (Shared configuration)
- **Issue:** The design doc specifies model `claude-sonnet-4-20250514` (or current Sonnet). The `AnthropicClient` hardcodes `claude-3-5-sonnet-20241022` as the default. This is likely a stale model string from an older Sonnet version. The discrepancy is between the design spec and the implementation; a caller who relies on the design spec to understand which model will be used will see different behaviour.
- **Suggested action:** Update the default model string in `client.ts` to match the design specification, or make the model a required configuration parameter with no default.

### W5: ADR-0009 and ADR-0010 are not referenced in the requirements appendix Decision Log

- **Location:** `docs/requirements/v1-requirements.md` Appendix: Decision Log vs `docs/adr/0009-test-diamond-strategy.md`, `docs/adr/0010-llm-response-validation-strategy.md`
- **Issue:** The requirements appendix lists ADRs 0001–0008. Two new ADRs have been written (ADR-0009: Test Diamond Strategy, ADR-0010: LLM Response Validation Strategy) but neither is referenced. ADR-0010 traces directly to Story 4.5 (LLM Error Handling) and Story 4.1 (Question Generation).
- **Suggested action:** Add ADR-0009 and ADR-0010 to the Decision Log appendix.

### W6: ADR-0004 status remains "Proposed" — not "Accepted"

- **Location:** `docs/adr/0004-roles-and-access-control-model.md` line 6
- **Issue:** ADR-0004 carries `**Status:** Proposed`. All other ADRs are `Accepted`. The decision has been fully implemented: `user_organisations` table, `is_org_admin()` function, `assessment_participants.contextual_role` column, and RLS policies all reference it.
- **Suggested action:** Update ADR-0004 status from `Proposed` to `Accepted`.
