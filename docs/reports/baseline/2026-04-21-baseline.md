## Summary

Since the 2026-04-18 baseline, three V2 epics have been fully delivered: E17 (Agentic Artefact Retrieval), E18 (Pipeline Observability & Recovery), and E19 (GitHub Issues as Artefact Source). V1 core coverage is 74% of in-scope stories. The biggest remaining gap is PRCC (Epic 2, 6%). The most significant divergence found in this scan: the "My Assessments" view shows all org assessments to every user instead of filtering to the current user's participant records — the participant-scoped and admin-wide views need to be split. Two critical discrepancies: the assessments view conflation and the missing `org_id` guard on `finalise_rubric`.

## Divergent (code differs from spec)

### Epic 5: Web App & Auth

#### Story 5.4 / Story 6.3: "My Assessments" vs. Org Admin overview conflated into one view
- **Status:** Divergent
- **Spec says:**
  - Story 5.4: "My Assessments" shows assessments where **the current user is a participant** (pending / completed).
  - Story 6.3: Org Admin sees a separate overview of **all** org assessments.
- **Code does:** The `/assessments` page (`src/app/(authenticated)/assessments/page.tsx` line 59–63) queries all assessments for the org scoped only by `org_id` — it does not filter by `assessment_participants.user_id`. Every authenticated org member sees all org assessments regardless of whether they are a participant. An Org Admin flag controls whether a "New Assessment" button is shown, but the list itself is identical for admins and non-admins.
- **Impact:** A regular user who has never been listed as a participant sees all org assessments. The "My Assessments" label is misleading. No separate admin overview (Story 6.3) exists. The two views (participant-scoped and admin-wide) need to be split.
- **Key files:** `src/app/(authenticated)/assessments/page.tsx:59–63`, `src/app/api/assessments/route.ts:52–58`
- **Recommendation:** Add a join to `assessment_participants` for non-admin users; provide a separate admin view for Story 6.3. RLS policy on `assessments` should also enforce this at the DB layer for non-admin reads.
