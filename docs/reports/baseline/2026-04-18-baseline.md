# Baseline: As-Built Requirements

**Date:** 2026-04-18
**Project phase (CLAUDE.md):** "Phase 1: Core Feature Implementation"
**Requirements versions reviewed:** v1 (v1.3), v2 (v0.4), v3 (v1.0), req-onboarding-and-auth, v1-prompt-changes, v2-proposed-additions

## Summary

The project has a fully functional FCS assessment engine (Epic 4: 100% delivered), working GitHub App integration for org/repo management, and a web application with auth, answering, and basic results. V3 enhancements (hints + configurable depth) are delivered. PRCC auto-trigger flow (Epic 2) and reporting dashboards (Epic 6) remain unbuilt. The biggest divergence is CLAUDE.md declaring "Phase 1" when significant Phase 2 work (web UI, results, onboarding auth) has already shipped. E11 (Artefact Quality Scoring) was cancelled and reverted on 2026-04-18.

## Delivered (matches spec)

### Epic 4: Shared Assessment Engine — `docs/requirements/v1-requirements.md`

#### Story 4.1: Question Generation from Artefacts
- **Status:** Delivered
- **Acceptance criteria met:** 3–5 questions generated, three Naur layers (world-to-program, design justification, modification capacity), `additional_context_suggestions` included as passive metadata, `artefact_quality: code_only` flag supported via `classifyArtefactQuality()`
- **Implementation:** `src/lib/engine/generation/generate-questions.ts`, `src/lib/engine/prompts/prompt-builder.ts`, `src/lib/engine/prompts/classify-quality.ts`

#### Story 4.2: Answer Scoring Against Rubric
- **Status:** Delivered
- **Acceptance criteria met:** LLM scores each answer 0.0–1.0, separate call per participant, score rationale captured
- **Implementation:** `src/lib/engine/scoring/score-answer.ts`

#### Story 4.3: Aggregate Score Calculation
- **Status:** Delivered
- **Acceptance criteria met:** Weighted sum across questions and participants, per-question aggregates calculated
- **Implementation:** `src/lib/engine/aggregate/calculate-aggregate.ts`

#### Story 4.4: Relevance Detection
- **Status:** Delivered
- **Acceptance criteria met:** Binary classification (relevant / not relevant)
- **Implementation:** `src/lib/engine/relevance/detect-relevance.ts`

#### Story 4.5: LLM Error Handling
- **Status:** Delivered
- **Acceptance criteria met:** Retry 3× with exponential backoff, `rubric_failed` state, Org Admin retry via `POST /api/assessments/[id]/retry-rubric`
- **Implementation:** `src/lib/engine/llm/client.ts` (OpenRouterClient), `src/app/api/assessments/[id]/retry-rubric/route.ts`

## Emergent Features (not in any requirement)

### Infrastructure
- **Structured logging (Pino):** `src/lib/logger.ts` — JSON structured logging with child loggers for request context. Matches cross-cutting concern in v1-requirements but not a standalone story.
- **API layer utilities:** `src/lib/api/` (auth.ts, context.ts, errors.ts, response.ts, validation.ts) — standardised auth extraction, error handling, Zod validation. Internal infrastructure.
- **Supabase client factories:** `src/lib/supabase/` (client.ts, route-handler.ts, route-handler-readonly.ts, secret.ts, server.ts) — multiple client patterns for different auth contexts. Internal infrastructure.
- **Sync debounce table:** `supabase/schemas/tables.sql` (sync_debounce) — webhook debounce queue for PR events. Forward-looking infrastructure for PRCC.
- **UI component library:** `src/components/ui/` (shadcn: button, card, badge, page-header, status-badge) — design system primitives.

### Organic additions
- **Comprehension depth + hints (V3):** Fully delivered but originated from V3 requirements created after V1 MVP. Requirements exist (v3-requirements.md) so technically covered, but shipped ahead of V1 completion. Back-specified.
- **FCS merged PRs tracking:** `fcs_merged_prs` table tracks which merged PRs compose an FCS assessment. Not explicitly a requirement story but necessary for the FCS creation flow.

### Orphaned code
- **None identified.** All code modules map to either a requirement, infrastructure need, or design decision.
