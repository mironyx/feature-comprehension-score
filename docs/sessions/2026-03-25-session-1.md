## Decisions Made

### Controller/service split

Route handlers in Next.js are HTTP entry points only. Business logic belongs in a service file.
The controller is ≤ 5 lines: call context factory, call service, return response.
This mirrors MVC: controller = thin dispatcher, service = business logic.

### `ApiContext` as per-request composition root

Next.js App Router has no DI container. `createApiContext(request)` assembles all infrastructure
clients per request from the incoming `NextRequest`. This is the composition root for each request.
Keeps DI possible without introducing a DI framework alien to the ecosystem.

### Parameter struct rule nuance

A struct/interface is acceptable when it represents a genuine domain concept (e.g. `FcsCreateInput`,
`AssessmentContext`). It is not acceptable when its only purpose is to avoid a long argument list
(e.g. `FetchContext { supabase, assessmentId, userId }`). The rule distinguishes intent.

### Existing violations are a follow-up item

`GET /api/assessments` and `GET /api/assessments/[id]` create clients inline without a service layer.
These predate the pattern and should be refactored in a follow-up issue. Noted in the LLD.
