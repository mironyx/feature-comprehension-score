### Post-PR refinement — LLD-wrong case (commit `ed450eb`)

User identified a gap: Agent C blindly trusts the LLD. If the LLD's internal decomposition
section itself is wrong, Agent C would either:
- Pass bad functions (they're in the wrong LLD), or
- Block good functions (they're not in the wrong LLD).

Resolution added to Step 4:

- **If LLD has no internal decomposition section** — do not block private helpers; emit
  "LLD gap — update the internal decomposition section" warn instead. Block only unspecified
  exported/public functions (public API surface should always be designed).
- **If LLD has an internal decomposition section but function is not in it** — block, but
  include both resolution paths in the evidence: (a) add `// Justification:` comment if the
  function is genuinely necessary, OR (b) update the LLD if the LLD was wrong. Added explicit
  note: "do NOT treat the LLD as infallible — the finding surfaces the gap; the resolution
  is a human decision."

## Decisions Made

### Agent C trusts neither the implementation nor the LLD

The key insight: when implementation and LLD disagree, either could be wrong. Agent C's job
is to surface the gap with both resolution paths visible. The human reviewer decides which
artefact is correct. This prevents Agent C from being a rubber stamp for a bad LLD.

### LLD completeness as a first-class check

Agent C first checks whether the LLD *has* an internal decomposition section before applying
blocking rules. An incomplete LLD is a different problem from an implemented-not-designed
function. The former requires an LLD update; the latter requires justification or removal.
