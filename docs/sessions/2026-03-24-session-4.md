## Decisions Made

### Service client for participant counts (key architectural decision)

The spec said participant counts come from "a subquery on `assessment_participants`". This is
incorrect: the `participants_select_own` RLS policy limits non-admin users to seeing only their
own row. A user-session query would return `participant_count: 1` for any non-admin user.

Resolution: `fetchParticipantCounts()` uses `createSecretSupabaseClient()` (service role, bypasses
RLS). Participant counts are non-sensitive aggregate metadata (just numbers). A regression test was
added to guard this decision permanently.

### ADR-0014: Inline contract types

The user raised that API routes were not self-contained — a reader had to open the design doc to
understand what the endpoint accepts and returns. Created ADR-0014 to require:
1. A JSDoc comment on the handler naming the HTTP method, path, all parameters, and response codes
2. Interfaces for each response body shape
3. `const body: ResponseType = {...}` so TypeScript enforces shape conformance

### `validateEnumParam` helper design

User requested a helper to avoid repeating the `if (raw && !VALID_SET.has(raw)) throw` pattern.
The null check lives inside the helper (not at the call site), so call sites read cleanly:
`validateEnumParam(typeRaw, VALID_TYPES, 'type filter')`.

### Cognitive complexity reduction

SonarQube (via diagnostics hook) flagged the `GET` handler at cognitive complexity 23 (threshold
15). Extracted `fetchParticipantCounts`, `toListItem`, and `validateEnumParam` as module-level
helpers, reducing it to 19. Residual complexity is inherent to the endpoint's single
responsibility (parse params + auth + query + map + respond).
