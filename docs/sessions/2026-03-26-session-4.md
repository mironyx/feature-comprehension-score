## Cost retrospective

Post-PR delta is small ($0.32) — one fix commit (review findings) and one CI fix commit (MD028). That is acceptable overhead.

**The headline cost ($2.26 for 2 tests + a lint fix) is high. Root causes:**

1. **Issue decomposition mismatch** — Issue #103 was created before PR #105 was written, and PR #105 already implemented the scoring trigger. By the time this session started, the "implementation" work was done; only test coverage remained. The issue was not re-scoped before `/feature` was called. The full session cost ($2.26) was spent reading context (LLD, service.ts, existing tests), understanding the overlap, then writing ~30 lines of test code.

   *Mitigation: before calling `/feature`, check if the issue's implementation is already partially done in a merged PR. If yes, re-scope the issue body to reflect what remains (tests only, integration only, etc.). A narrow issue = less context to read = lower cost.*

2. **Three-agent `/pr-review` is expensive relative to the diff size** — The diff was 32 lines. The review cost was ~$0.35 (estimated from delta). Three agents loading the full LLD to check a 32-line test diff is disproportionate.

   *Mitigation: for pure test-only PRs (no source files changed), consider skipping `/pr-review` or using a lighter single-agent pass. The value of the three-agent review is highest when implementation files change.*

3. **CI failure on a pre-existing lint error** — The MD028 fix required a third commit and push, which re-triggered CI and added minor overhead.

   *Mitigation: run `npx markdownlint-cli2 "**/*.md"` as part of Step 5 full verification, not just on changed files. The current `/feature` skill only lints changed files; a pre-existing error in an unchanged file will still fail CI.*

**Net improvement actions:**
- Add a note to `/feature` Step 1 pre-flight: check if implementation already exists in a merged PR before proceeding.
- For test-only PRs, consider a lighter review pass (single agent, no LLD read).
- Add full `markdownlint` scan to Step 5 verification to catch pre-existing violations before CI runs.
