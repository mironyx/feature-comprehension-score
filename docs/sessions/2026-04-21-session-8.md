## Decisions made

1. **Four-sub-step test-first flow followed to the letter** (interface → `test-author` → implementation → evaluator). The independent sub-agent enumerated 14 observable contract properties before any behaviour code was written. Implementation then made those tests pass without altering their assertions. `feature-evaluator` verdict: PASS WITH WARNINGS (5 adversarial tests — above the 3-test reporting threshold, logged below).
2. **Defence-in-depth on RLS.** The original LLD relied on `answers_select_own` alone; PR review caught that `policies.sql` OR's that policy with `answers_select_admin`, so an admin-who-is-also-a-participant viewer would otherwise have received every participant's answers and `MyScoresSection` would have rendered an arbitrary match as "Your score"/"Your answer". Fixed by threading the viewer's `participation_id` through `fetchMyAnswers` and adding an explicit `.eq('participant_id', participantId)` filter. Regression test asserts the filter pair is applied. LLD §3 updated to record the correction.
3. **Shared `HeaderSection`.** The feature name / repo / date / participant completion / depth badge header is identical across all three role branches; extracted into its own component rather than duplicating in each view. Added to LLD §3 internal decomposition.
4. **`toDecimalScore` helper.** Story 3.4 requires own scores in `0.00`–`1.00` form while aggregate scores stay in percentage form — a separate formatter keeps the intent obvious at each call site. Added to LLD §3 helpers.

## Cost retrospective

| Driver | Evidence | Impact |
|--------|----------|--------|
| **Context compaction** | Session summary reload before lld-sync step; turn-~125 pre-compact snapshot captured in draft | High — re-summarising the full session inflates cache-write tokens |
| **PR-review agent fan-out** | 3 parallel review agents (Quality, Design, Framework) each re-sent the full diff | Medium — `/pr-review-v2` adaptive path already caps at 2–3 agents; this diff was >150 lines so the full path was justified |
| **Fix cycle for the privacy-leak blocker** | One extra commit + regression test + full re-verify pass | Medium — avoidable with a tighter LLD (see next row) |
| **LLD under-specified the RLS surface** | §3 mentioned `answers_select_own` only; missed that it is OR'd with `answers_select_admin` | Medium — a 30-second grep of `policies.sql` during `/lld` would have caught this |
| **Evaluator volume** | 5 adversarial tests (above 3-test threshold) | Low — the extras were cheap and caught a real edge case (`—` fallback), but signal that the `test-author` prompt could enumerate cross-role absence properties on its own |

**Improvement actions for next time:**

1. **In `/lld`, cross-reference RLS policies whenever a helper says "RLS restricts this."** Policies can be OR'd — the code must not assume a single policy admits rows. A one-line grep check in the LLD template would have surfaced the OR'd admin policy before implementation started.
2. **Feed "cross-role absence" into the `test-author` prompt.** For any role-based feature, instruct the agent to enumerate not just what each role _sees_ but also what each role must _not_ see (the other roles' content). This would have absorbed 2 of the 5 evaluator adversarial tests into the property test.
3. **Keep the two-agent PR review path (no framework search) when the diff is entirely inside a single file hierarchy.** Agent B (framework best practices) added minimal value here because the change was UI-only with no new framework surface. A "framework changes detected" gate would skip Agent B cleanly when no `package.json`, config, or framework-import files change.
