### Post-review fixes (this session)

#### Design gap: unhandled throw from initial GitHub fetch

The `/pr-review` agent raised a bug: if `GET /user` or `GET /user/orgs` returned 5xx, `githubFetch` would throw and propagate uncaught out of `syncOrgMembership`. The auth callback's `try/catch` swallowed it silently.

Fix:
1. Wrapped initial `Promise.all` in try/catch inside `syncOrgMembership` ‚Äî treats network/GitHub failures as transient, preserves existing rows, returns.
2. Removed the outer `try/catch` from the auth callback ‚Äî `syncOrgMembership` is now explicitly no-throw by design.
3. Added 8th BDD test: "Given GitHub returns a server error on the initial user/orgs fetch, then existing memberships are preserved."

## Decisions Made

- **`syncOrgMembership` is no-throw** ‚Äî all GitHub and DB error paths preserve existing memberships and return rather than throwing. The auth callback does not need try/catch around it. This is a stronger contract than the original spec.
- **Live provider token, not stored token** ‚Äî the spec said "decrypted from `user_github_tokens`". The live session token is available at callback time and is the freshest credential. No decryption needed.
- **Cognitive complexity of `mergeRawArtefacts`** (S3776, complexity 24 > 15) ‚Äî left as-is. Reducing it would require extracting 7 single-purpose merge helpers. Documented as known trade-off; not worth the abstraction overhead at this stage.
- **Primitive Obsession in `artefact-source.ts`** ‚Äî `RepoCoords` already exists for the worst case. Remaining `string` parameters are standard GitHub API types. Not worth branding.
