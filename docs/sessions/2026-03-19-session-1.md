- Mock implementation suite: verifies the interface contract (single PR and multi-PR forwarding)

## Decisions Made

- Added `PRExtractionParamsSchema` (Zod) alongside the interface ‚Äî consistent with the rest of the engine layer, provides runtime validation for callers that parse external input.
- `z.number().int().positive()` for PR numbers: positive integers only (PR numbers start at 1). Slightly tighter than the design spec (`number[]`) but correct in practice.
- `.min(1)` on `prNumbers` array: prevents a degenerate call with no PRs.

### Issue 2: `gh pr merge --delete-branch` fails from a worktree

`gh pr merge --delete-branch` does three things locally after merging on GitHub:
1. Merges on GitHub ‚úÖ
2. `git checkout <base-branch>` locally ‚ùå ‚Äî fails: `fatal: 'main' is already used by worktree`
3. `git branch -d <feature-branch>` ‚ùå ‚Äî never reached

The merge succeeds on GitHub but local cleanup fails. A second `gh pr merge` attempt from the primary worktree then hits "already merged".

Root cause: `gh pr merge` was run from inside the feature worktree while `main` was already checked out in the primary worktree. `gh` attempts a local checkout of `main` as part of `--delete-branch` cleanup, which git refuses.

**Fix applied:** Updated `.claude/skills/feature-end/SKILL.md` step 4 to:
- Check PR state first ‚Äî if already `MERGED` (user merged via GitHub UI), skip the merge command entirely.
- Document that `gh pr merge` must be run from the primary worktree, not from inside the feature worktree.

### Issue 3: `claude_session_feature` textfile never reached Prometheus

Three sub-problems found and fixed:

1. **Missing `node_exporter` container** ‚Äî `monitoring/docker-compose.yml` had no `node_exporter`. Added `prom/node-exporter` with `--collector.disable-defaults` and `--collector.textfile`, mounting `./textfile_collector:/textfiles:ro`.
2. **Missing scrape target** ‚Äî `monitoring/prometheus.yml` only scraped `otel-collector`. Added `node-exporter:9100` as a second scrape target.
3. **CRLF line endings** ‚Äî Python on Windows writes `\r\n` by default; the Prometheus textfile parser requires `\n`. Fixed the session tagging script in `/feature` skill to use `newline="\n"` on `write_text()`. Rewrote the existing `.prom` file with correct line endings.

**Verified:** `claude_session_feature{session_id="a3a44e00...", feature_id="FCS-46"}` now appears in Prometheus with correct labels. Session‚Üífeature join queries will work from the next session onwards.
