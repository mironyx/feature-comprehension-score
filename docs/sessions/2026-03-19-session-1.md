### Implementation

Created `src/lib/engine/ports/artefact-source.ts` — a pure port interface (14 lines):
- `ArtefactSource` interface with `extractFromPRs(params: PRExtractionParams): Promise<RawArtefactSet>`
- `PRExtractionParams` type and `PRExtractionParamsSchema` (Zod) for runtime validation
- No framework imports; depends only on `RawArtefactSet` from the engine prompts layer

Created `tests/lib/engine/ports/artefact-source.test.ts` — 10 tests across two suites:
- Mock implementation suite: verifies the interface contract (single PR and multi-PR forwarding)
- Schema validation suite: validates all field constraints (owner/repo non-empty, prNumbers positive integers and non-empty)

## Decisions Made

- Added `PRExtractionParamsSchema` (Zod) alongside the interface — consistent with the rest of the engine layer, provides runtime validation for callers that parse external input.
- `z.number().int().positive()` for PR numbers: positive integers only (PR numbers start at 1). Slightly tighter than the design spec (`number[]`) but correct in practice.
- `.min(1)` on `prNumbers` array: prevents a degenerate call with no PRs.

## Review Feedback Addressed

Two weak tests identified during review and fixed:

1. **Removed** redundant schema test `'validates params with multiple PR numbers for FCS'` — it exercised the same schema rule as the single-PR test with no additional coverage.
2. **Replaced** `'then it can accept multiple PR numbers for FCS assessment'` — the original test asserted on a hardcoded mock return value, not on the params passed through. Replaced with `'then it forwards all PR numbers to the implementation unchanged'`, which uses `toHaveBeenCalledWith(expect.objectContaining(...))` to verify the params contract.
