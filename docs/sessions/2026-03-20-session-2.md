## Decisions made

### Distinct UUID prefix for auth users

The original LLD spec assigned auth users UUIDs in the `00000000-0000-0000-0000-*` range — the same as organisations. Although there is no database conflict (different tables), the same UUID value appearing in two different entities causes confusion when reading error output or test failures. Auth user UUIDs were changed to use `a0000000-0000-0000-0000-*` prefix. The UUID scheme was also extended to all seed entities (repos `0001-*`, assessments `0002-*`, etc.) so any UUID in a log can be immediately identified by its segment value.

### `resetDatabase` uses cascade DELETE, not TRUNCATE

The LLD spec described `TRUNCATE ... CASCADE` in dependency order. The implementation uses cascade `DELETE` starting from root tables (`user_github_tokens` then `organisations`). Both approaches produce the same end state; `DELETE` with cascade is simpler to implement correctly — no need to enumerate leaf tables in the right order.

### `resetDatabase` does not touch auth.users

The spec said auth users should be re-seeded after truncation. In practice, auth user lifecycle is managed separately via `createTestUser`/`deleteTestUser` (already in `tests/helpers/supabase.ts`). Keeping `resetDatabase` scoped to public schema tables is simpler and avoids coupling test reset logic to the Supabase Auth admin API.

### `fileParallelism: false` in vitest config

Integration test files share a single local Supabase instance. Without serial file execution, the `resetDatabase` test and the seed data verification tests interfere when run concurrently. `fileParallelism: false` serialises file-level execution. Unit tests are unaffected in terms of correctness; the suite takes slightly longer (~20s total vs ~10s) but remains well within acceptable bounds.

### Factory signatures use `Partial<XRow>` overrides, not named option interfaces

The LLD spec showed named option interfaces (`CreateOrgOptions { name?, status?, githubOrgId? }`). The existing factories in the codebase already used `Partial<XRow>` from the generated database types — a pattern that avoids maintaining a parallel interface layer. The new `createTestAnswer` factory uses a structured `CreateTestAnswerOptions` because it has four required foreign-key fields.
