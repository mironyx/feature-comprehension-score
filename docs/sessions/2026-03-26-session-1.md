## Decisions made

- **`ApiContext` omits `githubClient`** — the spec included `githubClient: Octokit` upfront, but assessment routes don't need GitHub access. Adding it would wire a redundant client on every request. Deferred; will be added when a GitHub-dependent route is implemented.
- **`validateSubmission` uses derived args** — spec passed the raw `ParticipantRow`; implementation passes `isFirstAttempt` and `previouslyIrrelevantIds` (derived before calling). Keeps validation pure.
- **`finaliseSubmission` takes `llmClient`** — spec had 3 args. Fourth arg added because scoring requires the LLM client and a single instance must be shared across the request.
- **`scoreAssessment()` doesn't exist** — spec named this as a single engine call. No such function exists; implementation calls `scoreAnswers()` + `calculateAssessmentAggregate()` from the pipeline, wrapped in a `triggerScoring()` service function.
- **Relevance write-back step added** — spec flow didn't mention a separate UPDATE after relevance checks. Added to persist `is_relevant` and `relevance_explanation` on the stored rows. Non-fatal; failures logged with inline comment.
- **`AnswerResult.attempts_remaining`** — added to response shape to inform client of remaining retries on `relevance_failed`.
