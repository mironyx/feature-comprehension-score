## Decisions made

### `forbidden()` for Server Component page protection (not `requireOrgAdmin()`)

The LLD specified using `requireOrgAdmin()` from §2.4 for all admin route protection. During
implementation it became clear `requireOrgAdmin()` takes `NextRequest`, making it usable only
in API route handlers — not Server Component pages. For pages, Next.js 15's `forbidden()` from
`next/navigation` is the correct mechanism (returns a genuine HTTP 403 response). Enabled via
`experimental.authInterrupts: true` in `next.config.ts`. LLD §2.6 updated to document this
distinction.

### `fetchOrgContext` parallel query helper

The layout needs both the current org record and the user's membership list before it can render
the NavBar. These two queries are independent, so they run in parallel via `Promise.all`. A
third query fetches other-org memberships only when needed (non-empty `otherOrgIds`). Extracted
into a private `fetchOrgContext` helper to keep the layout function under the 25-line limit.

### Inline admin check in `organisation/page.tsx`

The page performs its own admin check (re-queries `user_organisations`) rather than relying
solely on the layout's `isAdmin` context. This is the correct pattern for defence-in-depth:
the layout hides the link, but the page enforces the policy at the HTTP level. This matches
the "each route validates independently" principle in the LLD.

### `ForbiddenPage()` replaced with `forbidden()` (review fix)

Initial implementation returned `ForbiddenPage()` (a JSX render returning HTTP 200). The
`/pr-review` agent flagged this as a blocker — the acceptance criterion requires HTTP 403.
Fixed by switching to `forbidden()` which triggers Next.js's 403 interrupt mechanism.
