### Summary

Fixed a bug where users who installed the GitHub App on their personal account (account type `User`) could not see their account in the org-select UI after sign-in.

**Root cause:** `syncOrgMembership` built the DB query using only org IDs from `GET /user/orgs`. GitHub's `/user/orgs` endpoint returns GitHub Organisations only â€” never personal accounts. The personal account was correctly stored in `organisations` (by the webhook handler) but never matched, leaving `user_organisations` empty.

**Fix (two changes in `src/lib/supabase/org-sync.ts`):**

1. Include `githubUser.id` in the IDs used for the `organisations` DB query:
   ```ts
   const githubAccountIds = [githubUser.id, ...githubOrgs.map((o) => o.id)];
   ```
2. Short-circuit the org membership API call for personal accounts (`org.github_org_id === githubUser.id`), defaulting role to `'member'`. The `/orgs/:login/memberships/:username` endpoint does not exist for personal accounts.

## Decisions made

- **Personal account detection:** Identify personal accounts by `org.github_org_id === githubUser.id` rather than adding an `account_type` column to the `organisations` table. The webhook handler already stores the GitHub user ID as `github_org_id` for personal accounts, so this is sufficient without a schema change.
- **Default role for personal accounts:** `'member'` â€” there is no membership API for personal accounts, and the owner of a personal account is always effectively a member.
