## Decisions Made

### `my_scores` split to dedicated endpoint

Mixing the detail fetch (always needed, fast) with self-view score retrieval (FCS-only,
post-submission, involves multi-attempt answer processing) was driving complexity in a single
route handler. The concerns are genuinely separate use cases. Splitting them is the correct design.
See design doc §2.4 for the full rationale. New issue #95 created for `GET /api/assessments/[id]/scores`.

### `filterQuestionFields` in `helpers.ts`

Next.js App Router only permits HTTP method exports from `route.ts`. Any other export causes a
build failure. Extracted to `[id]/helpers.ts` so tests can import it directly.

### Explicit field projection in `filterQuestionFields`

The LLD spec used `...q` spread. Implementation uses explicit field listing to prevent inadvertent
exposure of new DB columns added in future migrations — consistent with ADR-0005.

### CodeScene thresholds enforced via targeted extraction

Rather than ignoring CodeScene warnings (which the user explicitly rejected), extracted the minimum
set of helpers to bring each function under the cc < 10 and LoC < 50 thresholds. Helpers are
private (not exported) and named for the operation they perform, not for their position in the
calling code.
