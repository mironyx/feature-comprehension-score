## Decisions made

### Subagents (Agent tool) over agent teams
Agent teams are experimental, require separate OS processes (Prometheus port conflict), and split-pane mode is unsupported in VS Code. Subagents via the Agent tool run within the same OS process, share the Prometheus endpoint, and support `isolation: "worktree"` for git safety.

### Prometheus over OTLP for local dev
Single OS process (subagent model) means one Prometheus endpoint suffices. OTLP + collector (Docker) is only needed if running multiple OS-level sessions. Keeping Prometheus avoids the Docker dependency for metrics.

### Single `/feature` skill, not a separate `/features`
Extending `/feature` with multi-issue argument parsing is cleaner and backward compatible:
- `/feature` ‚Üí top 1, single agent
- `/feature 123` ‚Üí specific issue, single agent
- `/feature 123 456 789` ‚Üí parallel subagents
- `/feature -n 3` ‚Üí top N Todo items, parallel subagents

### OTLP push over Prometheus pull (late decision)

After session close, decided to switch from Prometheus pull to OTLP push:
- Pull model requires Prometheus to reach each dev's `localhost:9464` ‚Äî blocked by VPNs, firewalls, home networks on a team
- Push model: all sessions push to a stable endpoint; solo uses local Docker collector, team just changes one env var
- Local overhead: one extra container (OTel collector) + 20-line config ‚Äî acceptable
- `settings.json` still set to `prometheus` ‚Äî needs updating next session
