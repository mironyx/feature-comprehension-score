### 2. Playwright configuration (`playwright.config.ts`)

Created a Playwright configuration targeting our Next.js app:

- **Browser:** Chromium only (lightweight; Firefox/WebKit can be added later).
- **Test directory:** `tests/e2e/` with `*.e2e.ts` naming convention — keeps E2E tests separate from Vitest unit/integration tests.
- **Base URL:** `http://localhost:3000`.
- **Web server auto-start:** the config uses `process.env.CI` to split behaviour:
  - **Locally:** runs `npm run build && npm run start` (full build then serve).
  - **In CI:** runs `npm run start` only (the CI pipeline already has a separate build step, so no need to double-build).
  - Timeout set to 120s to accommodate WSL2/CI cold-start latency.
- **Reporters:** HTML report locally (for interactive debugging), list reporter in CI (for clean log output).
- **Retries:** 0 locally, 1 in CI (catches flaky infrastructure, not flaky tests).
- **`reuseExistingServer`:** enabled locally so you can keep a dev server running; disabled in CI for clean runs.

### 4. MSW handler scaffolds

Created a layered MSW architecture for mocking external APIs (GitHub and Anthropic Claude) in Vitest tests:

**`tests/mocks/github.ts`** — Three handler factory functions:

- `mockPullRequest(owner, repo, prNumber, overrides)` — mocks `GET /repos/:owner/:repo/pulls/:number`. Returns configurable PR data (number, title, state, merged, user, base/head refs). Supports per-test overrides via spread.
- `mockCheckRuns(owner, repo, ref, overrides)` — mocks `GET /repos/:owner/:repo/commits/:ref/check-runs`. Returns check run status/conclusion.
- `mockOrgMembership(org, username, role)` — mocks `GET /orgs/:org/members/:username`. Returns membership role.

**`tests/mocks/anthropic.ts`** — Two handler factory functions:

- `mockClaudeMessages(responseContent, overrides)` — mocks `POST /v1/messages`. Returns a well-formed Claude API response with configurable text content, model, usage stats. Used for question generation, scoring, and relevance assessment mocks.
- `mockClaudeError(status, message)` — mocks `POST /v1/messages` with an error response. Used for testing error handling paths (rate limits, auth failures, etc.).

**`tests/mocks/handlers.ts`** — Default handler set that aggregates a baseline GitHub PR mock and a baseline Claude response mock. Individual tests override these via `server.use()`.

**`tests/mocks/server.ts`** — MSW `setupServer()` instance configured with default handlers. Imported by test files that need HTTP mocking.

**Design rationale:** factory functions over static JSON fixtures. This gives tests fine-grained control — a test for "PR with failed checks" just calls `mockCheckRuns(owner, repo, ref, { check_runs: [{ conclusion: 'failure' }] })` rather than maintaining a separate fixture file. The `server.use()` + `server.resetHandlers()` pattern ensures test isolation.

### WSL2 Chromium dependency note

Playwright's `npx playwright install chromium` downloads the browser binary but not the system-level shared libraries it depends on. In CI, `--with-deps` handles this automatically. Locally on WSL2 (Ubuntu 24.04), the `t64` package name transition means some libraries have new names. The working install command:

```bash
sudo apt-get install -y libnspr4 libnss3 libatk1.0-0t64 libatk-bridge2.0-0t64 \
  libcups2t64 libdrm2 libxkbcommon0 libxcomposite1 libxdamage1 libxfixes3 \
  libxrandr2 libgbm1 libpango-1.0-0 libcairo2 libasound2t64 libatspi2.0-0t64
```

Note: the original `libasound2` package name no longer works on Ubuntu 24.04 — it must be `libasound2t64`. Similarly for `libatk1.0-0`, `libatk-bridge2.0-0`, `libcups2`, and `libatspi2.0-0`.

## Decisions

1. **Playwright web server command split by environment** — locally runs `build && start`; in CI runs `start` only (CI build job runs separately before the E2E job). This avoids double-building and keeps CI fast.

2. **Playwright test agents initialised** — set up via `npx playwright init-agents --loop=claude` (introduced in Playwright v1.56, article reference: [Playwright test agents on Medium](https://medium.com/@dneprokos/how-playwright-test-agents-are-changing-the-game-in-e2e-automation-5827e19574ae)). The command initially hung because it waits for a TTY confirmation prompt — piping input (`echo "n" | npx playwright init-agents --loop=claude`) resolved this. The command generated six files:
   - **`.mcp.json`** — registers a Playwright MCP server (`playwright run-test-mcp-server`) exposing browser control as MCP tools (click, type, navigate, snapshot, etc.).
   - **`.claude/agents/playwright-test-planner.md`** — Planner agent definition. Uses MCP browser tools to explore the live app, map user flows, and output structured Markdown test plans to `specs/`.
   - **`.claude/agents/playwright-test-generator.md`** — Generator agent. Reads a plan from `specs/`, replays each step live in the browser, reads the execution log, then writes a `.spec.ts` file.
   - **`.claude/agents/playwright-test-healer.md`** — Healer agent. Runs failing tests, pauses on errors, inspects the page via snapshots/selectors, edits test code, and re-runs until green (or marks as `test.fixme()`).
   - **`tests/e2e/seed.spec.ts`** — Seed test that bootstraps the environment for agent-generated tests (navigates to home page, asserts title).
   - **`specs/README.md`** — Directory for Markdown test plans generated by the Planner.

   Key insight: these agents are Claude Code subagent definitions, not a standalone Playwright feature. Playwright exposes browser actions as MCP tools, and the agent markdown files define how Claude Code should orchestrate them. This is why VS Code 1.105+ or Claude Code is required as the "loop" host.

3. **MSW handler factories over static fixtures** — handler factories with per-test overrides via `server.use()` are more flexible than maintaining separate JSON fixture files per scenario. Tests compose the exact response shape they need, and `server.resetHandlers()` in `afterEach` ensures isolation.

4. **BDD naming convention** — all tests use nested `describe`/`test` with Given/When/Then phrasing, matching the project's established convention from the Supabase integration tests.
