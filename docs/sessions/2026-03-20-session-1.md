## Decisions made

### Explicit constraint names (correction to LLD spec)

The LLD spec did not name the updated constraints. During implementation, the auto-generated UNIQUE constraint name would have truncated `is_reassessment` to `is_reassessm` (PostgreSQL 63-char limit), making it unreliable to reference in future migrations. Both constraints were given short explicit names:

- `chk_answers_attempt_number` (CHECK)
- `uq_answers_participant_question_reassessment` (UNIQUE)

Rule going forward: always use explicit constraint names when column name combinations exceed ~35 characters.

### Story 3.6 (re-assessment) deferred post-MVP

After reviewing the requirements and discussing the design, Story 3.6 (`POST /api/assessments/[id]/reassess`) has been deferred. The `is_reassessment` column is retained as scaffolding, but the business logic and endpoint are out of scope until requirements are revisited.

The core tension: the current design models re-assessment as a personal confidence exercise (frozen team aggregate, only the individual sees updated scores). However, the more valuable use case may be team-level remediation — which would require a new assessment with a link back to the original, not rows appended to the existing one. This needs requirements clarification before implementation.

See LLD §2.1 implementation notes for details.

### Constraint logic clarification

The `chk_answers_attempt_number` constraint enforces:
- Original answers (`is_reassessment = false`): attempt 1–3 only (re-answer flow for irrelevant answers)
- Re-assessment answers (`is_reassessment = true`): no upper limit (`attempt_number` acts as a cycle counter)
