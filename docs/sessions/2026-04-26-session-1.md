## Decisions made

- **Deviated from LLD T3 colour values to satisfy WCAG AA.** The LLD prescribed `--color-accent: #d97706` and claimed 4.6:1 contrast on `#f5f4f0`. Actual ratio is 2.89:1 — well below the 4.5:1 acceptance criterion. Same problem for destructive (`#dc2626` = 4.35:1) and success (`#16a34a` = 2.97:1). Shipped:
  - `--color-accent`: `#92400e` (amber-800, 6.3:1)
  - `--color-accent-hover`: `#78350f` (amber-900, preserves darken-on-hover)
  - `--color-destructive`: `#b91c1c` (red-700, 6.0:1)
  - `--color-success`: `#15803d` (green-700, 4.56:1 — close to the threshold; flagged in docs and LLD)
- **Test pattern: file-read regex.** Matched the existing `tests/app/assessments/answering-form-styling.test.ts` style — read `globals.css` once, run `RegExp` assertions per variable. Avoids needing a CSSOM or jsdom-style runtime parse for what is fundamentally a static-asset check.
- **Did not introduce any TypeScript theme switcher or `dark:` Tailwind variants.** This task is tokens-only; T4 will deliver the runtime toggle. The contract rule in `frontend-system.md` was tightened explicitly to prevent later tasks from sneaking `dark:` variants in.
