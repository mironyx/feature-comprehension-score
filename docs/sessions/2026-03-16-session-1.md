### Issue #31: Assessment pipeline orchestrator

- **Branch:** `feat/assessment-pipeline-orchestrator`
- **PR:** #44 (targeting `feat/assessment-engine`)

Implemented the pipeline orchestrator that wires all Phase 1 engine components into a complete assessment flow: generate rubric ‚Üí score answers ‚Üí calculate aggregate.

**Source files:**

- `src/lib/engine/pipeline/assess-pipeline.ts` ‚Äî three public functions:
  - `generateRubric()` ‚Äî delegates to question generation, wraps result in Rubric type
  - `scoreAnswers()` ‚Äî relevance-first flow per answer (skip scoring for irrelevant), handles partial failures
  - `calculateAssessmentAggregate()` ‚Äî computes overall, per-participant (weighted), and per-question (unweighted) scores
- `src/lib/engine/pipeline/index.ts` ‚Äî barrel exports

**Test file:**

- `tests/lib/engine/pipeline/assess-pipeline.test.ts` ‚Äî 4 BDD tests:
  1. Valid artefacts ‚Üí generates rubric with questions, weights, reference answers
  2. Rubric + 2 participants' answers ‚Üí scores all, correct aggregate
  3. One LLM call fails ‚Üí records failure, continues scoring remaining answers
  4. Generation fails ‚Üí returns generation_failed status

## Review feedback addressed

- **Relevance-first flow:** Refactored from parallel score+relevance to sequential relevance‚Üíscore. If answer is irrelevant, skip the scoring LLM call entirely (saves cost).
- **CodeScene Large Method:** Extracted `processAnswer()` helper from `scoreAnswers()`.
- **CodeScene Excess Arguments:** Bundled LLM params into `LLMCallConfig` interface.
- **Type consistency:** Extracted `ScoringFailure` named interface (was inline anonymous type in `failures` array, inconsistent with `ScoredAnswer`).
- **Comments:** Added JSDoc to aggregate section explaining the three score levels and why question-level uses unweighted mean.
- **Test names:** Fixed misleading "fails after retries" ‚Äî pipeline doesn't retry, that's the LLM client's responsibility.
- **Test simplification:** Replaced manual `callCount` state machine with Vitest's `mockResolvedValueOnce` chaining.
- **VS Code launch.json:** Fixed debug configuration ‚Äî was trying to run test file directly with Node, replaced with proper Vitest debugger config.

## Decisions made

- Pipeline processes answers sequentially (not concurrent) ‚Äî bounded concurrency is a future optimisation requiring its own tests and rate limit handling
- Irrelevant answers get score 0 and are excluded from aggregation ‚Äî they don't penalise participants
- `ScoringFailure` extracted as named type for consistency with `ScoredAnswer`
- Question-level scores use unweighted mean (topic comprehension signal, not biased by importance weight)
