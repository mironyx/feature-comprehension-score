## Decisions made

- **`SignInButton.tsx` split from `page.tsx`** — Next.js App Router requires `'use client'` for interactive elements; the server component handles the `?error` query param (a Promise in Next.js 15 that must be `await`ed server-side).
- **`storeProviderToken` not extracted as helper** — single call site; no reuse benefit. Inlined in the GET handler.
- **`/` is protected, not public** — spec listed `/` as public. Correct behaviour is to protect it — unauthenticated requests redirect to sign-in. Matcher pattern excludes only `auth/` and `api/webhooks/` prefixes.
- **Org membership sync deferred** — callback spec included org sync as step 5. Deferred to §2.3 / future issue. Redirect hardcoded to `/assessments` (no `returnTo` support yet).
- **`output: standalone` + `next start` incompatibility** — `next start` does not work with standalone output. `npm run start` changed to `node .next/standalone/server.js`. E2E tests now rely on `.env.local` for local env vars rather than inline env block in playwright config.
- **ci-probe `permissionMode: bypassPermissions`** — project-level `permissions.allow` in `settings.json` does not apply to background subagents. The `permissionMode` frontmatter field in the agent definition is the correct mechanism.

## Post-merge fixes (same session)

After PR #79 was merged, two process bugs were fixed and pushed directly to `main`:

### `gh-project-status.sh` — broken board item lookup (commit `d8c516a`)

`find_item_id` used `gh project item-list --format json` and matched on `content.number`, but the
GitHub CLI does not include `content.number` in that output (all items returned `None`). Every
board status update during `/feature-end` therefore failed with "issue not found on project board".

Fix: replaced `gh project item-list` with a `gh api graphql` query that fetches `projectItems`
directly from the issue node, filtering by project ID. This is reliable regardless of CLI version.

### `/feature-end` skill — too many Bash approval prompts (commit `d8c516a`)

Steps 5 and 6 (branch delete, board update, issue close) were three separate Bash calls, each
requiring user approval. Merged into a single chained call with `2>&1; true` guards so any
already-complete step is silently skipped.
