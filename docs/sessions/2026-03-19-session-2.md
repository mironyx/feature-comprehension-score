### Design review — context file resolution bug

Reviewing `GitHubArtefactSource.extractFromPRs` revealed three compounding design errors in the
context file fetch:

1. **Wrong SHA** — code used `perPR[0]!.headSha` (first PR's feature branch HEAD). The LLD said
   "target branch HEAD" (`main`). These are different refs; `main` is correct for stable docs.
2. **Wrong config scope** — LLD referenced `repositories.context_file_patterns`. The actual table is
   `repository_config` (separate from `repositories`). Schema already correct; docs were wrong.
3. **Incomplete multi-PR scan** — only the first PR was used for context file resolution. For FCS
   assessments, all PRs must be scanned; a PR that updates a design doc should supply its version.

### ADR 0013 — Context file resolution strategy

Created `docs/adr/0013-context-file-resolution-strategy.md` documenting three options and the
accepted decision:

- **Config scope:** `repository_config.context_file_patterns` overrides `org_config.context_file_patterns`
  (`NULL` = use org default). Repo-level override is necessary because repos within the same org
  have different doc structures.
- **Baseline ref:** target branch HEAD (`main`), not any PR's feature branch HEAD.
- **PR-specific override:** if a PR modifies a file matching a context pattern, use the PR version
  instead of the `main` baseline. For multi-PR FCS, all PRs are scanned; last PR by merge date wins.

### Code fix — `GitHubArtefactSource`

Fixed the three context file errors and improved code quality:

- `PRExtractionParamsSchema`: added `defaultBranch: z.string().min(1).optional()`
- `RepoCoords` interface introduced — groups `owner`/`repo` across all private methods (addresses
  Primitive Obsession and String Heavy Function Arguments diagnostics)
- `GlobMatcher` type introduced — compiled glob matcher passed instead of raw `string[]` (avoids
  primitive obsession on the context pattern parameter)
- `extractSinglePR`: now detects changed files matching context patterns, fetches them at PR HEAD —
  these become PR-specific overrides in the merged result
- `extractFromPRs`: baseline fetched at `params.defaultBranch ?? 'main'`; PR-specific overrides
  applied on top (baseline first, PR version wins by path)
- `mergeRawArtefacts`: added `context_files` deduplication (last PR wins, consistent with
  `file_contents` merge strategy)
- `fetchContextOverrides` 5-arg method eliminated — replaced by pure helper `filterByGlobs` +
  reuse of `fetchFileContents`

## Decisions Made

- Leave REST API (no GraphQL refactor) — 3 parallel REST calls per PR is acceptable; diff requires
  REST anyway; GraphQL optimisation deferred to V2.
- `defaultBranch` is optional in `PRExtractionParams` with adapter default `'main'` — avoids
  breaking callers that don't know the default branch, while allowing explicit override.
- "PR override wins" semantics over "supplement" — avoids duplicate content; the PR version is
  always more relevant for assessment purposes.
