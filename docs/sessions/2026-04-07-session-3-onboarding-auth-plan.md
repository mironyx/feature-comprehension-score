## Decisions

- **No member-list mirror table.** ADR-0020 §6 rewritten. Requirements and ADR
  were contradicting each other; requirements (the simpler path) wins. Revisit
  only if per-sign-in API latency becomes measurable.
- **No feature flag on the sign-in cutover.** Nothing is deployed yet, so the
  dual-write / cutover / cleanup sequence collapses into one atomic task. Task
  7 (drop `user_github_tokens`) merges into Task 3.
- **Task 5 (webhooks) parallelisable** with Tasks 2–3 under `/feature-team` —
  no file overlap between webhook handlers and the sign-in path.
- **Old phase LLDs are not retroactively rewritten.** Per ADR-0018,
  `lld-phase-2-web-auth-db.md` §2.3 is superseded in place by the new epic's
  per-task LLDs. ADR-0020 §Neutral updated to match.
- **`/create-plan` now mirrors `/architect`** in shape: both take an argument,
  both produce epic/task-structured output, both name their successor.

## Insights

- **Requirements vs ADR drift is real.** The requirements doc explicitly said
  "no mirror tables" in Out of Scope, while ADR-0020 §6 called for the mirror.
  Neither author (me) noticed at write time. The drift only surfaced when I
  ran the `/create-plan` decomposition and had to choose which doc to
  implement against. **Lesson:** a `/drift-scan` between requirements and ADR
  before planning would have caught this sooner. Worth considering as a
  preconditional step for `/create-plan` when the input references an ADR.
- **The `/create-plan` → `/architect` chain had a broken first link.**
  `/create-plan` had no arg handling while every other skill in the chain
  did. Small friction, compounding — exactly the kind of thing that only
  surfaces when you try to use the workflow end-to-end rather than reading
  the skill docs in isolation.
- **The harness article's "guides not knowledge" framing is exactly right
  here.** The fix wasn't adding content to `/create-plan` — it was aligning
  its *shape* with the downstream skills. The guide is a protocol between
  skills, not a document.
