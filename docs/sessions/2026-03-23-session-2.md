### Bug fixes

#### 1. Browser client uses implicit flow instead of PKCE (`src/lib/supabase/client.ts`)

`createClient` from `@supabase/supabase-js` defaults to implicit flow. Supabase redirects back
with `#access_token=...` in the URL fragment, which is never sent to the server. The
`/auth/callback` route handler received no `code` parameter.

Fixed: switched to `createBrowserClient` from `@supabase/ssr`, which uses PKCE and sends
`?code=...` as a query parameter.

#### 2. Dynamic `process.env[key]` doesn't work client-side (`src/lib/supabase/env.ts`)

Next.js replaces `NEXT_PUBLIC_` vars at compile time via static property access only.
`process.env[key]` with a variable resolves to `undefined` in the browser bundle.

Fixed: use `process.env.NEXT_PUBLIC_SUPABASE_URL` and
`process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` directly; key name passed as a label
argument for the error message only.

#### 3. OAuth scopes must be explicit in client code (`src/app/auth/sign-in/SignInButton.tsx`)

GitHub returns 403 on `GET /user/emails` without the `user:email` scope explicitly requested.
Dashboard configuration alone is insufficient.

Fixed: added `scopes: 'user:email read:user'` to `signInWithOAuth` options.

### pgsodium on Supabase cloud — known limitation

`store_github_token` (SECURITY DEFINER, runs as `postgres`) calls
`pgsodium.crypto_aead_det_encrypt`. On Supabase cloud:

- pgsodium extension is NOT enabled by default — must be enabled via migration or dashboard.
- `postgres` role does NOT have execute permission on the crypto functions.
- `GRANT EXECUTE ON FUNCTION pgsodium.crypto_aead_det_encrypt TO postgres` fails even in
  migrations — the function is owned by the system superuser; `postgres` lacks grant option.

New migration `20260323153000_pgsodium_extension_and_grants.sql` enables the extension and
creates `github_token_key` idempotently. The grant problem remains unresolved.

Options under investigation:
- Migrate to **Supabase Vault** (`vault.create_secret()` / `vault.decrypt_secret()`) which is
  designed for this use case and is accessible to `postgres`.
- Contact Supabase support for superuser grant.

### Design doc updated

`docs/design/lld-phase-2-web-auth-db.md` annotated with three post-implementation corrections
covering the PKCE client requirement, OAuth scope requirement, and pgsodium cloud limitations.

## Decisions Made

- **`createBrowserClient` over `createClient`** — the LLD did not specify this; the browser
  client file predated the PKCE requirement. Now explicit in the design doc.
- **Scopes in client code** — dashboard-only scope configuration is unreliable; explicit scopes
  in `signInWithOAuth` are the correct pattern going forward.
- **pgsodium grant deferred** — token storage non-functional on cloud is acceptable at this
  stage (no user data at risk). Will be resolved before any real users sign in.
